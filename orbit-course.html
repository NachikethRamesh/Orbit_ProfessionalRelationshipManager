What Orbit Actually Does

A personal relationship manager that lives on your computer — not in the cloud

The Problem It Solves

You meet dozens of people — at conferences, over email, in meetings. Over time, relationships go cold. You forget to follow up. You lose track of who said what.

Orbit watches your email and calendar, figures out who matters, and tells you when to reach out — automatically.

📥

Auto-Import

Pulls contacts from Gmail, Calendar, and Google Contacts — no manual data entry.

🤖

AI Summaries

Reads your emails and meetings, then writes a one-sentence summary of each.

🌡️

Warmth Scores

Calculates a 0-100 "warmth" score for each relationship that decays over time.

💡

Smart Nudges

Suggests when to follow up, send a thank-you, or check in on someone.

A Day in the Life

Imagine you open Orbit on a Monday morning. Here's what happens when you click "Analyze":

4

What Makes It Special: Local-First

Unlike most apps, Orbit runs entirely on your computer. No cloud servers. No monthly subscriptions. Your data never leaves your machine.

💡
Key Insight: Local-First Software

Most apps today send your data to a company's servers. "Local-first" means the app stores everything on YOUR computer using a database file. No internet required, no data privacy concerns, and it works forever — even if the company disappears.

How the Code Is Organized

Orbit's codebase follows a layered structure — like a building with distinct floors.

src/ All the source code
app/ Pages the user sees + API routes
(app)/ Dashboard, Contacts, Reminders, Settings pages
api/ Backend endpoints — the "phone lines" the frontend calls
services/ Business logic — syncing, summarizing, scoring, chatting
components/ Reusable UI building blocks (cards, lists, modals)
lib/ Shared utilities — database connection, encryption, email filtering
hooks/ Data-fetching helpers that connect the UI to the API
💡
Key Insight: Separation of Concerns

Notice how the code is split into layers — pages, API routes, services, and utilities. Each layer has one job. This pattern — called "separation of concerns" — makes it easy to change one part without breaking everything else. When talking to an AI coding agent, you can say "put this logic in the service layer, not the API route" and it knows exactly what you mean.

The Real Code Behind Step 1: Filtering Junk

Before Orbit processes any email, it filters out the noise. Newsletters, automated receipts, and bot messages get thrown out. Here's how:

CODE 
const BLOCKED_PREFIXES = [
  "noreply", "no-reply", "no_reply",
  "donotreply", "do-not-reply",
  "mailer-daemon", "postmaster",
  "notifications", "newsletter",
  "unsubscribe", "receipts",
];
PLAIN ENGLISH

Create a list of email name patterns that are definitely not real people.

If the part before the @ symbol matches any of these — like "noreply@company.com" — the email gets skipped.

This is the first line of defense. It catches the obvious junk before the AI even sees it.

But what about borderline cases — like "team@startup.com" which could be a person or a mailing list? For those, Orbit sends a batch of up to 80 email addresses to GPT-4o-mini and asks: "Which of these are real people?"

Graceful Failure: The Safety Net

Here's something clever about this pipeline — every step is wrapped in its own safety net. If Gmail sync fails, Calendar sync still runs. If AI summarization breaks for one email, the others still get processed.

CODE 
try {
  const gmailResult = await syncGmail(userId, account.id);
  gmail = gmailResult.syncedCount;
} catch (err) {
  console.error(`Gmail sync failed:`, err);
}
PLAIN ENGLISH

Try to sync Gmail and count how many new emails we got.

Save the count so we can report it later.

If anything goes wrong, just log the error and keep going.

Don't crash. Don't stop the whole pipeline. Just note the problem and move on.

🔑
Why This Matters for You

When directing an AI to build a multi-step process, always ask for try/catch around each step. Say: "Each step should fail independently — don't let one broken step crash the whole pipeline." This is the difference between an app that's fragile and one that's resilient.

The AI Brain

How Orbit uses two different AI models — and why it picks each one

Two Models, Two Jobs

Orbit doesn't just throw everything at one AI model. It uses the right tool for the job — like hiring a junior assistant for simple tasks and a senior advisor for complex ones.

⚡

GPT-4o-mini

Fast and cheap. Used for simple tasks: summarizing emails, classifying whether an address is a real person. Runs hundreds of times per sync.

🧠

GPT-4o

Powerful and thoughtful. Used for complex reasoning: analyzing relationships, generating follow-up suggestions, powering the chat. Runs a few times per sync.

💡
Key Insight: Model Selection Matters

Using the biggest AI model for everything is like taking a helicopter to the grocery store — expensive and unnecessary. Smart apps pick the cheapest model that's good enough for each task. When building with AI, always ask: "Does this task need deep reasoning, or just pattern matching?" Use the smaller model when you can, the bigger one when you must.

How the Summarizer Works

Every email and meeting gets a one-sentence summary. Here's the actual code that talks to the AI:

CODE 
const response = await getOpenAI().chat.completions.create({
  model: "gpt-4o-mini",
  messages: [
    { role: "system", content: systemPrompt },
    { role: "user", content: userMessage },
  ],
  max_tokens: 150,
  temperature: 0.3,
});
PLAIN ENGLISH

Connect to OpenAI and ask the AI to generate a response...

Use the fast, cheap model (gpt-4o-mini) — good enough for summaries.

Send two messages: a "system" message that defines the AI's personality and rules...

...and a "user" message with the actual email content to summarize.

Limit the response to 150 tokens (about 2 sentences).

Set temperature to 0.3 — keep it factual and consistent, not creative.

How Action Items Are Generated

This is where the bigger AI model shines. It looks at your recent interactions with someone and decides: should you follow up? Send a thank-you? Check in?

CODE 
const systemPrompt = `You are a proactive PRM assistant.
Return a JSON array of action items.
Each item must have:
  "type": "follow_up" | "thank_you" | "check_in"
  "title": "Short action title"
  "body": "Why this action is recommended"
  "priority": 1-5`;
PLAIN ENGLISH

This is the "instruction manual" given to the AI before it sees any data.

It tells the AI: "You're a relationship assistant. Your job is to suggest actions."

It specifies the exact format the response must follow — JSON with specific fields.

By defining the format in advance, the app can reliably parse the AI's response without guessing.

🔑
Prompt Engineering in Action

Notice how the system prompt tells the AI exactly what format to use. This is called "structured output" — you tell the AI to return data in a specific shape so your code can reliably read it. When building AI features, always define the output format in the system prompt.

Warmth Scores & Auto-Reminders

How Orbit tracks relationship health and knows when to nudge you

The Warmth Score: A Relationship Thermometer

Imagine a campfire. If you keep adding wood (interactions), it stays warm. Stop feeding it, and it slowly dies out. That's exactly how warmth scores work — every interaction adds heat, and time makes it fade.

CODE 
const TYPE_WEIGHTS: Record<string, number> = {
  meeting: 3,
  email_sent: 2,
  email_received: 1,
};

const DECAY_WINDOW_MS = 90 * 24 * 60 * 60 * 1000;
PLAIN ENGLISH

Different interactions are worth different amounts:

A face-to-face meeting is worth 3 points (most valuable)

Sending an email is worth 2 points (you took initiative)

Receiving an email is worth 1 point (they reached out)

The "decay window" is 90 days — interactions older than that don't count at all.

The Freshness Formula

Here's the core algorithm — the math that turns interaction history into a single number:

CODE 
for (const interaction of interactionList) {
  const weight = TYPE_WEIGHTS[interaction.type] ?? 1;
  const ageMs = now - occurredAt;
  const freshness = Math.max(0, 1 - ageMs / DECAY_WINDOW_MS);
  totalScore += weight * 10 * freshness;
}
return Math.min(100, Math.round(totalScore));
PLAIN ENGLISH

Go through every interaction with this contact...

Look up how much this type of interaction is worth (3 for meetings, 2 for sent emails, 1 for received).

Calculate how old this interaction is.

Calculate "freshness" — a meeting yesterday has freshness near 1.0, a meeting 80 days ago has freshness near 0.1.

Add: weight × 10 × freshness to the running total. A recent meeting adds ~30 points. An old email adds ~2.

Cap the final score at 100. Nobody can score higher than 100.

Auto-Reminders: The Smart Nudge System

Orbit doesn't just score relationships — it acts on those scores. Four types of automatic reminders:

[cold] Warmth below 20 — "Reach out to Sarah — your relationship is cooling down"
[no-contact] No interaction for 30+ days — "You haven't talked to Mike in over a month"
[follow-up] Received email with no reply — "Follow up on Alex's message about the project"
[meeting] Recent meeting occurred — "Debrief: your meeting with the design team"

The clever part? Reminders auto-dismiss. If you email someone after getting a "[cold]" reminder, Orbit marks it as completed automatically. It adapts to what you actually do.

💡
Key Insight: Time-Decaying Scores

This pattern — giving more weight to recent events and letting old ones fade — appears everywhere in software. Search engines use it (recent pages rank higher), recommendation systems use it (recent purchases matter more), and social feeds use it (newer posts appear first). The formal name is "exponential decay." When you want your AI app to prioritize recent data, ask for "time-weighted scoring with decay."

Security: Keeping Secrets Safe

How Orbit protects your Google tokens with military-grade encryption

The Problem: Storing Passwords

When you connect your Google account, Google gives Orbit a special key (called a token) that lets it read your email and calendar. If someone got that key, they could read all your emails.

Orbit can't just store these tokens in plain text — that would be like writing your house key code on the front door. Instead, it encrypts them.

How Encryption Works: The Lockbox

Think of encryption like a lockbox with a combination. You put a letter inside, scramble the lock, and only someone with the combination can open it. Here's the actual code:

CODE 
const iv = crypto.randomBytes(12);
const cipher = crypto.createCipheriv(
  "aes-256-gcm", key, iv
);
let encrypted = cipher.update(plaintext, "utf8", "hex");
encrypted += cipher.final("hex");
const authTag = cipher.getAuthTag();
return `${iv}:${authTag}:${encrypted}`;
PLAIN ENGLISH

Generate 12 random bytes — like spinning the lockbox dial to a random starting position. This ensures encrypting the same text twice gives different results.

Create the encryption tool using AES-256-GCM — the same standard used by banks and governments.

Feed in the secret text (the Google token) and get back scrambled gibberish.

Finish the encryption process.

Get the "authentication tag" — this proves the data hasn't been tampered with later.

Pack everything together: random-start-position : tamper-check : scrambled-data

The Full Security Model

🔐

Tokens Encrypted at Rest

Google tokens are encrypted before being saved. Even if someone copies your database file, they can't read the tokens without the encryption key.

🏠

Localhost Only

Orbit only listens on your own computer (localhost:3000). No one on the internet can connect to it.

📁

User-Only File Permissions

The config file (~/.orbit/.env) is created with permissions that only your user account can read.

💡
Key Insight: Defense in Depth

Good security isn't one big wall — it's many layers, like an onion. Each layer protects against a different threat. Encryption protects the database file. Localhost protects the network. File permissions protect the config. If one layer fails, the others still hold. When asking AI to build secure features, say: "I want defense in depth — encrypt sensitive data, restrict network access, and limit file permissions."
